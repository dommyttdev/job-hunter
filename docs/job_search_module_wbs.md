# Job Search RSS Module WBS

## 前提

- 対象は `docs/job_search_module_design.md` に記載された Job Search RSS Module のMVPです。
- MVPの対象求人サイトは atGP のみです。
- dodaチャレンジ、Web UI、ユーザーアカウント、複数サイト間の求人名寄せ、高度なランキング、求人本文の詳細差分表示は対象外です。
- ここで扱う「購読条件登録」は、ユーザーアカウントに紐づく購読管理ではなく、RSSフィードを生成するための条件定義を保存する機能です。
- 期間や担当者の割り当ては含めず、TDDで進めるための作業順と細分化を目的とします。

## TDD進行方針

- 各機能は「失敗するテストを書く」「最小実装で通す」「リファクタリングする」の順で進めます。
- 最初は受け入れレベルの薄い縦断テストを用意し、fake実装で通してから、外側の実装へ差し替えます。
- 外部サイトアクセス、DB、RSS出力、スケジューラは中核ユースケースの後にアダプタとして実装します。
- RSS生成時に求人サイトへアクセスしないこと、収集失敗時に削除検知しないことは、早い段階からテストで固定します。
- MVPのE2Eは最後にまとめて作るのではなく、薄い縦断スライスから段階的に太くします。
- 仕様調査や設計整理はテストコード化できる期待値に落としてから実装へ進みます。
- 自動テストは実サイトの状態に依存させず、atGPの実アクセス確認は任意の手動スモーク確認として分離します。

## Definition of Done

- 各機能作業は、先に失敗するテストがあり、そのテストを通す最小実装とリファクタリングが完了していること。
- ユースケース層のテストは、FastAPI、SQLAlchemy、HTTPX、Playwright、atGP固有実装に依存していないこと。
- RSS生成のテストでは、求人サイトアクセスが発生しないことを確認していること。
- 購読条件は、APIまたは手動コマンドから登録でき、収集条件へ反映されること。
- 削除検知のテストでは、正常収集完了時だけ削除扱いが発生することを確認していること。
- atGPアダプタは、保存HTML fixtureと期待データに対するテストで主要な変換を確認していること。
- 自動テストは実サイトアクセスを前提にせず、実サイト確認は手動スモーク手順として分離していること。
- Repository実装は、fakeとSQLite実装の両方で同じ契約テストを通していること。
- MVP完了時には、fixtureベースの縦断テスト、単体テスト、統合テスト、lintが通っていること。

## 主要マイルストーン相当の完了単位

期間は定義しませんが、TDDでの作業完了のまとまりとしては以下の順序で確認できます。

1. テスト基盤があり、最初のテストを実行できる
2. ドメインモデルが外部依存なしの単体テストで保証されている
3. インメモリ実装で購読条件、収集条件、差分検出、RSS生成が動く
4. atGPアダプタが保存HTML fixtureに対するテストで保証されている
5. SQLAlchemy/SQLite実装がRepository契約テストを通る
6. fixtureベースの薄いE2Eで、購読からRSS生成まで一連で通る
7. FastAPI、購読条件登録、スケジューラ、手動実行入口を含めてMVPとして動かせる
