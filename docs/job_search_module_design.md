# Job Search RSS Module Design

## Purpose

このモジュールの目的は、求人サイト上の求人情報を定期的に収集し、求人の変化をRSSとして配信することです。

RSSでは、利用者が以下の条件を指定して求人の更新情報を購読できるようにします。

- 地域: 都道府県、市区町村
- 職種: 職種カテゴリ、職種詳細
- 地域と職種の組み合わせ

RSSに載せる対象は単なる検索結果ではなく、前回収集時点からの変化です。

- 新規掲載された求人
- 内容が更新された求人
- 掲載終了または検索結果から消えた求人

最初の対応サイトはatGPとし、MVPでは地域検索、職種検索、地域と職種を組み合わせた検索に対応します。

現在想定している対象サイトは以下です。

- atGP: https://www.atgp.jp/search/top/search_result
- dodaチャレンジ: https://doda.jp/challenge/kyujin/

ただし、MVPで実装する対象は引き続きatGPのみとします。dodaチャレンジは将来追加候補として扱います。

## Design Principles

### RSS配信を中心に設計する

このモジュールの主目的は検索URLを作ることではなく、求人の変化をRSSとして安定して配信することです。

検索URL生成、求人一覧取得、詳細ページ取得、パース処理は、RSS配信のための内部処理として扱います。外部利用者にとって重要なのは「どの条件で、どの求人に、どんな変化があったか」です。

### RSS読み取り時に求人サイトへアクセスしない

RSS生成時に求人サイトへアクセスすると、RSSリーダーのアクセス頻度がそのまま求人サイトへの負荷になります。

そのため、求人サイトへのアクセスは定期収集処理に限定します。RSS生成は、保存済みの求人スナップショットと差分情報だけを参照します。

### 購読された条件だけを収集対象にする

全地域、全職種の組み合わせを網羅的に収集するのは非効率で、求人サイトにもシステムにも負荷が高くなります。

そのため、RSS購読として登録された条件を収集対象にします。利用者が購読していない地域・職種条件は、原則として収集しません。

購読条件と実際の収集条件は分離します。購読条件は利用者向けのRSSフィルタであり、収集条件は求人サイトへアクセスする単位です。

MVPでは、正規化した条件から同一条件を判定し、同じ条件を複数の利用者が購読していても収集は1回だけ行います。将来的には、条件の包含関係、購読数、RSSの最終アクセス時刻などをもとに、収集条件や収集頻度を最適化できる構造にします。

### 変化を一次データとして保存する

求人の現在状態だけでなく、求人に発生した変化を明示的に保存します。

これにより、RSS生成時に毎回差分計算を行わずに済みます。また、あとから「いつ新規掲載されたか」「いつ更新されたか」「いつ消えたか」を追跡できます。

### サイト固有仕様を外側に閉じ込める

求人サイトごとに、検索パラメータ、HTML構造、求人IDの取り方、マスターデータの表現は異なります。

これらの差異はサイト別アダプタに閉じ込めます。収集、差分検出、RSS生成といった中核処理は、atGPなど特定サイトの仕様を知らない状態に保ちます。

### 複数サイト追加に備えて疎結合にする

新しい求人サイトを追加するときに、既存の収集処理、差分検出、RSS生成を大きく変更しない構造にします。

サイトごとの差異はアダプタとマッピングに閉じ込め、共通処理は抽象化されたインターフェースだけに依存させます。これにより、サイト追加時の変更範囲を局所化し、既存サイトへの影響を抑えます。

### DIで依存関係を外から組み立てる

中核処理が、具体的なサイト実装、HTTPクライアント、データベース実装、RSSレンダラーを直接生成しないようにします。

これらの具体実装はDIで外から渡します。ユースケースは抽象に依存し、実行環境やテストでは必要な実装を差し替えられるようにします。

### クリーンアーキテクチャを基本方針にする

設計はクリーンアーキテクチャの考え方に基づけます。

求人収集、差分検出、RSS生成といったアプリケーションの中核ルールを内側に置き、サイト別実装、DB、HTTP、RSS出力などの詳細は外側に置きます。

依存方向は外側から内側へ向けます。中核のユースケースやドメインモデルが、atGP、SQLAlchemy、requestsなどの具体技術へ依存しない状態を保ちます。

### プロジェクト構成は責務ごとに分離する

プロジェクト構成は、技術単位だけでなく責務の境界が分かる形にします。

少なくとも、ドメイン、ユースケース、外部インターフェース、サイト別アダプタ、永続化、RSS出力は分離します。これにより、どこに何を実装するべきかを明確にし、サイト追加やテスト追加を容易にします。

### 共通モデルは求人配信に必要な情報へ寄せる

内部の共通モデルは、各求人サイトの画面構造ではなく、RSS配信とフィルタリングに必要な情報を中心に定義します。

代表的には以下を扱います。

- 求人を一意に識別する情報
- タイトル、会社名、詳細URL
- 勤務地、職種、雇用形態、給与などRSS本文に必要な情報
- 地域・職種で絞り込むための正規化情報
- 内容変更を検出するためのハッシュや更新判定材料

サイト固有の生データは、永続化前に共通モデルへ変換します。

### 地域・職種は正規化して扱う

RSS利用者には、サイト固有の検索パラメータではなく、都道府県、市区町村、職種カテゴリ、職種詳細といった意味のある条件を指定させます。

そのために、サイト別のマスターデータと共通的な地域・職種概念を対応付けて管理します。将来ほかの求人サイトを追加しても、利用者側の指定方法はできるだけ変えません。

### 削除検知は「収集条件」と結びつけて考える

求人が検索結果から消えた理由は、掲載終了、条件変更、サイト側の表示変更など複数あります。

MVPでは、ある収集条件で以前見えていた求人が、正常に完了した次回収集で見えなくなった場合に「削除または掲載終了」として扱います。

この判断は収集条件に依存するため、どの条件で収集した結果なのかを記録します。

### 収集処理と配信処理を分離する

収集処理は外部サイトアクセス、HTML解析、差分検出を担当します。

配信処理は保存済みの差分を読み、RSSとして整形するだけにします。

この分離により、収集頻度、RSS配信頻度、エラー処理、再実行方針をそれぞれ独立して設計できます。

### 失敗時に過剰な状態変更をしない

求人サイトへのアクセス失敗、HTML構造変更、途中ページの取得失敗が起きた場合、不完全な収集結果をもとに大量の削除扱いを発生させてはいけません。

削除検知は、収集対象の検索結果を正常に取得できた場合だけ実行します。収集失敗は状態変更ではなく、失敗した実行履歴として記録します。

### 追加サイトに備えるが、MVPを複雑にしすぎない

将来的に複数の求人サイトを扱える構造にします。ただし、MVP時点ではatGP対応に必要な抽象だけを導入します。

サイト追加時に変わってよいものはサイト別アダプタとマッピング設定です。収集、差分検出、RSS生成の中核ユースケースは、できるだけ変更しない方針にします。

### データベースはサイト共通構造を優先する

MVPではサイト別テーブルを作らず、求人、求人変更、地域、職種、サイト別マッピングを共通構造で扱います。

サイト固有の追加情報が必要になった場合は、まず汎用的な属性として保持できないかを検討します。サイト別テーブルは、共通構造では表現できない要件が明確になってから導入します。

### データモデルも疎結合かつ正規化する

データモデルは、特定サイトのHTML構造や検索パラメータに引きずられないようにします。

求人、地域、職種、サイト別マッピング、求人変更履歴を分けて保持し、RSS配信や検索条件に必要な意味単位で正規化します。サイト固有値はマッピングとして扱い、中核の求人データや変更履歴から切り離します。

これにより、サイト追加、マスター変更、RSSフィルタ変更が互いに過度に影響しない構造にします。

## Main Responsibilities

このモジュールは大きく以下の責務を持ちます。

- RSS購読条件の登録と正規化
- 購読条件に基づく収集対象条件の管理
- サイト別マスターデータの収集と正規化
- 指定条件に基づく求人検索結果の収集
- 求人詳細情報の取得と共通モデルへの変換
- 前回スナップショットとの差分検出
- 新規、更新、削除の変更履歴保存
- 地域・職種・変更種別による変更履歴の抽出
- RSS XMLの生成

## Technology Stack

MVPでは、軽量に始めつつ将来の拡張に耐えられるPython中心の構成を採用します。

- 言語: Python
- Web/RSS配信: FastAPI
- DB: SQLite
- ORM: SQLAlchemy
- マイグレーション: Alembic
- HTTPクライアント: HTTPX
- JavaScript実行を伴う収集: Playwright
- HTML解析: BeautifulSoup4、lxml
- スケジューラ: APScheduler
- RSS生成: feedgen、または標準ライブラリによる専用Renderer
- テスト: pytest
- Lint/Format: ruff

MVPではSQLiteを採用します。単一環境での小規模収集を前提に、運用を軽く保つためです。将来的に複数ワーカー、並列収集、長期運用、監視要件が強くなった場合はPostgreSQLへの移行を検討します。

JavaScript実行が必要な求人サイト収集にはPlaywrightを採用します。SeleniumではなくPlaywrightを選ぶ理由は、自動待機、ブラウザコンテキストによる状態分離、将来の並列収集への拡張性が今回の用途に合うためです。

FastAPI、SQLAlchemy、HTTPX、Playwright、RSS生成処理は、中核ユースケースから直接扱いません。クリーンアーキテクチャの方針に従い、外部技術はアダプタやインフラ層に閉じ込め、ユースケースは抽象に依存させます。

## atGP MVP Scope

MVPではatGPを対象に、以下に限定して実装します。

- 地域マスターの取得
- 職種マスターの取得
- 地域条件による求人収集
- 職種条件による求人収集
- 地域と職種を組み合わせた求人収集
- 同一購読条件の重複収集回避
- 求人の新規、更新、削除検知
- 条件指定されたRSS生成

atGP固有の検索パラメータ、マスター構造、HTML構造、正規化ルールはatGP用アダプタ内に閉じ込めます。

dodaチャレンジはMVPでは実装しません。ただし、将来追加時に中核処理を変更せずに済むよう、atGP実装時点からサイト別アダプタと共通ユースケースの境界を意識します。

## Out of Scope for MVP

MVPでは以下を対象外とします。

- 複数求人サイト間での同一求人の名寄せ
- ユーザーアカウントや購読管理
- Web UI
- 高度なランキングやおすすめ
- 求人本文の詳細な差分表示
- サイト別に最適化された個別RSS項目

これらは、RSS配信の基本フローが安定してから追加を検討します。

## Testing Policy

テストでは、特に以下を重視します。

- サイト固有パーサーが期待どおり共通モデルへ変換できること
- 同じ求人を再収集しても不要な変更が発生しないこと
- 内容が変わった求人だけ更新として記録されること
- 正常な収集完了時だけ削除検知が行われること
- 地域・職種・変更種別のRSSフィルタが正しく働くこと
- RSS生成が外部サイトアクセスに依存しないこと

## Future Site Addition

新しい求人サイトを追加するときは、まずそのサイト固有の検索条件、マスター構造、求人一覧、求人詳細の取得方法をアダプタとして実装します。

共通の収集、差分検出、RSS生成の考え方を変えずに追加できるかを確認し、どうしても共通化できない要件が出た場合だけ中核モデルを見直します。
